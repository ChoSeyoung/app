# Developer Collaboration Guide

목적: 여러 개발자가 같은 기준으로 화면, 데이터, 문서를 수정하도록 공통 작업 규칙을 정리한다.

## 1. 작업 시작 전 확인 순서

1. `AGENTS.md`
2. `docs/design-system.md`
3. `docs/language-guide.md`
4. `docs/git-commit-policy.md`
5. 현재 작업 대상 화면/도메인 코드

UI 작업이라면 디자인 시스템을 먼저 보고, 문구 수정이라면 언어 가이드를 먼저 본다.  
커밋 전에는 커밋 메시지 정책을 다시 확인한다.

## 2. 화면 파일 작업 규칙

- 모든 `tsx` 파일은 파일 상단에 한국어 블록 주석을 둔다.
- 상단 주석은 최소한 아래 내용을 포함한다.
  - 역할
  - 유지보수 포인트
- 화면 루트는 `SafeAreaView`를 사용한다.
- 새 화면을 만들 때는 가능한 한 `components/design-system/*` 공통 컴포넌트를 먼저 검토한다.
- 화면에서 쓰는 문자열은 하드코딩하지 않고 `constants/i18n.ts`에 추가한다.

권장 예시:

```tsx
/**
 * 홈 탭 메인 화면.
 *
 * 역할:
 * - 오늘 식단과 기록 현황을 보여준다.
 *
 * 유지보수 포인트:
 * - 홈은 첫 행동 유도 흐름을 가장 먼저 고려한다.
 */
```

## 3. 주석 작성 규칙

- 주석은 한국어로 작성한다.
- 코드가 무엇을 하는지보다, 왜 이 구조로 유지해야 하는지를 설명하는 쪽이 더 중요하다.
- 의미가 자명한 한 줄 코드마다 주석을 붙이지 않는다.
- 화면 파일은 상단 헤더 주석, 복잡한 로직은 필요한 블록 주석 정도로 제한한다.

좋은 예시:

- `추천 이유 문구와 엔진 제약은 항상 같이 바뀌어야 한다.`
- `하단 고정 버튼이 있는 화면은 스크롤 하단 패딩도 함께 조정한다.`

지양 예시:

- `버튼을 누르면 저장한다.`
- `값을 state에 넣는다.`

## 4. UI/디자인 협업 규칙

- 색, 타이포, radius, spacing은 `docs/design-system.md`와 `constants/theme.ts`, `constants/spacing.ts`를 우선한다.
- 카드 간 세로 간격은 임의 수치 대신 spacing 토큰을 쓴다.
- 배경, 장식 버블, 토스트, 탭바처럼 공통 인상이 큰 요소는 개별 화면에서 임의로 변형하지 않는다.
- 디자인 시스템과 다른 결정이 필요하면 먼저 문서를 갱신한 뒤 구현한다.

## 5. 상태와 데이터 작업 규칙

- 화면 파일에서 데이터 저장 형식을 직접 만들지 않는다.
- 도메인 모델은 `features/*/model.ts`, 저장/조회는 `features/*/repository.ts`를 우선 사용한다.
- 로컬 저장 포맷이 바뀌면 import/export, 마이그레이션, 관련 화면을 함께 점검한다.
- 추천/집계 로직은 화면 안에 길게 넣지 말고 `features/*` 쪽으로 분리한다.

## 6. 문구와 다국어 규칙

- 새 문구는 `ko`, `en`를 함께 추가한다.
- 사용자 노출 문구 톤은 `docs/language-guide.md`를 따른다.
- 짧고 차분한 표현을 우선한다.
- 토스트 메시지는 제목 없이 한 줄 메시지를 기본으로 유지한다.

## 7. 라우팅 규칙

- 새 페이지를 추가하면 `app/_layout.tsx` 또는 `app/(tabs)/_layout.tsx` 등록 여부를 같이 점검한다.
- 탭 전용 화면인지, 스택 단독 화면인지 먼저 정하고 나서 경로를 만든다.
- 더보기/설정성 페이지는 탭 자체보다 개별 스택 화면으로 분리하는 편을 우선한다.

## 8. 검증 규칙

- 코드 수정 후 기본 검증은 `npm run lint`다.
- 화면 구조를 크게 바꿨다면 관련 화면 진입 경로를 함께 점검한다.
- 실제 디바이스 의존 기능은 아래를 따로 확인한다.
  - 알림: dev build 또는 standalone
  - 광고: Expo Go 제외 런타임
  - 백업/복원: 파일 왕복 테스트

## 9. 커밋 규칙

- 커밋 메시지는 `type(scope): summary`
- `type`, `scope`는 영문
- `summary`는 한국어

예시:

```text
feat(meal-plan): 식단 개인화 기준 편집 화면을 추가
fix(journey): 기록 상세 팝업 닫힘 플래시를 수정
design(common-ui): 플로팅 탭바 정렬과 여백을 조정
docs(git): 커밋 메시지 언어 규칙을 명확히 정리
```

## 10. 리뷰 체크리스트

- 이 변경이 디자인 시스템과 충돌하지 않는가?
- 문구가 `i18n`에 들어가 있는가?
- 같은 책임의 로직이 화면과 저장소에 중복되지 않는가?
- spacing과 radius가 토큰을 따르는가?
- 새로운 페이지가 적절한 라우터에 등록됐는가?
- 주석이 파일 역할과 유지보수 포인트를 설명하는가?

## 11. 로컬 이미지 자산 작업 규칙

- 식재료/레시피 실사 이미지는 수동 파일 추가보다 `docs/local-image-generation.md`의 로컬 생성 파이프라인을 우선 사용한다.
- 새 이미지 생성 후에는 반드시 `npm run image:sync`로 매핑 파일을 갱신한다.
- 생성 이미지 파일명은 `recipeId`, `ingredientId`와 정확히 일치시킨다.
- 플레이스홀더 UI를 지우기 전에 실제 에셋 매핑이 앱에서 정상 노출되는지 먼저 확인한다.
